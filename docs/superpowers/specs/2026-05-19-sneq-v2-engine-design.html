02TL;DR

Six things to hold in your head as you scroll the rest.

Single TS package @sneq/engine. Consumer owns the GM loop; engine owns facts, entities, constraints, propagation.

v2 Multi-campaign

One Engine instance manages many campaigns. engine.campaign(id).… facade, campaignId on every record.

v2 Layered entity resolution

alias → vector → LLM judge → user-prompt. Solves the "is this the same blacksmith we met 3 sessions ago?" gap in v1.

v2 Provider router

Three tiers — heavy, light, embeddings — each with primary + fallback chain. Quota-aware. No OpenAI, no Grok.

v2 Tool-call protocol

JSON Schema + Zod tool definitions. GM emits tool calls; consumer dispatches via handleToolCall().

deferred Pre-gen & bindings

v1's real-time pre-generation lives behind a no-op hook. TTRPG app and Hermes bindings get their own future specs.

03Why V2

v1 is conceptually solid but written for a real-time RPG-Maker-style consumer. Jean's real targets are turn-based dialog.

v1 framing

Real-time RPG. Sub-200ms cache hits. Elaborate pre-generation: predictor + workers + priority queue (chapter 5). Optimized for a player who can't wait.

Missing. Cross-session entity resolution. How does the engine know "the blacksmith" in session 7 is the same Aldric from session 4? No machinery for it.

v2 reframing

Turn-based dialog. 2–5s LLM latency tolerated. Pre-gen deferred behind an extension hook. Engine optimizes for correctness and persistence, not latency.

Adds. Layered entity resolver. Provider router with quota-aware fallback. Tool-call protocol for GM-to-engine fact ingestion. Multi-campaign scoping.

Stack rule · No OpenAI, no xAI/Grok in any default chain — carried from ~/.claude/Contexts/about-me.md.

08Router & Providers

Three task tiers, each with a primary and an ordered fallback list. Triggers: 429 · 401/403 · 5xx · timeout · malformed.

Disambiguation · Groq the chip company (inference platform) is fine. Grok the xAI model is banned. The router supports the former via openai-compatible with Groq's base URL.

09Tool Call Protocol

Tool schemas exported as Tool objects, Zod, and JSON Schema. Consumer wires them into the LLM call; engine dispatches via handleToolCall().

read

  • sneq__lookup_entity resolve a mention to an existing entity
  • sneq__get_entity full entity + figed attributes
  • sneq__get_relevant_facts facts for an entity, optionally by depth
  • sneq__suggest_existing generation-direction resolver

register

  • sneq__mention_entity introduces or re-uses an entity; returns isNew
  • sneq__register_fact appends a figed attribute · returns contradictions[] instead of throwing
  • sneq__add_constraint adds a CP constraint

collapse

  • sneq__collapse_attribute engine drives an LLM call (heavy tier) to fill a specific attribute, validates, inscribes, propagates

scene / turn

  • sneq__set_scene declares current location + present entities
  • sneq__advance_turn bumps the monotonic turn counter
Deliberate split · register_fact never throws on contradiction. Engine validates, consumer's GM adjudicates (regen · reinterpret · reject). Honors the bookkeeping-library contract.

Key contract

  • transaction() mandatory · appendFact + propagation must be atomic
  • upsertEntity · keyed by (campaignId, id)
  • searchEntitiesByVector · campaign + type filtered ANN
  • findEntitiesByAlias · normalized lookup, layer 1
  • neighbors(depth) · GCN traversal for propagation
Embedding dimension is fixed per file. vec0 pins D at table creation. All campaigns in one SQLite file must use the same embedding model. Mixing models = use separate Repository instances.
Out of scope for V2 · Convex, Postgres, Redis adapters. Interface is the contract; future contributions can follow.

16Open Questions

None blocking V2 — tracked so V2 doesn't accidentally close them off.

Q1 Embedding interoperability

Switching embedding providers mid-campaign breaks vector comparability. V2 doesn't migrate — embeddings must be regenerated. Migration tool is future work.

Q2 Resolver cold-start

At campaign start, vector search has no data; L2 always misses. Acceptable — L1 covers explicit names, L3 rarely needed early.

Q3 Propagation depth

v1 recommends 2–3 hops with decay. V2 keeps the default but exposes depth via EngineConfig. Defaults stand until evidence says otherwise.

Q4 Authority weighting

GM-attributed vs dice-attributed claims both go through registerFact with different Observation.source. V2 treats them equally in validation; future may weight dice as more authoritative.
